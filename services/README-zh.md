# 微服务套件（Microservice Suite） - 中文使用指南

**你现在看到的是我们共同构建的核心学习与实验项目** —— 一个**生产级质量的 8 服务微服务套件**。

它是用 TypeScript + Bun 运行时开发的，完全使用内存存储（不需要安装数据库），包含 **516 个测试全部通过**。既是高质量的学习参考资料，也是我们 Agent 进化框架（Hermes、self-evolution-coordinator 等）的可靠实验场，同时把之前的哲学思考（L0 人类完整性锚点）落地成了“可验证、可重复、不会失控”的具体代码。

### 这个东西到底是什么？能怎么用？

#### 1. **作为学习微服务的最佳参考（推荐入门方式）**
- **真实业务场景模拟**：像一个简化版的电商/ SaaS 系统。
  - **auth**：用户注册、登录、JWT token、黑名单、刷新令牌。
  - **users**：用户资料管理、角色权限。
  - **products**：商品目录、库存管理（支持批量操作、软删除）。
  - **orders**：订单全生命周期 + **严格有限状态机**（不能乱跳状态）。
  - **payments**：支付处理、退款、金额门槛规则（>1万会故意失败，用于测试）。
  - **notifications**：多渠道通知（邮件、短信、推送），支持模板和投递追踪。
  - **analytics**：事件追踪、漏斗分析、cohort 留存、 retention 指标。
  - **search**：基于 TF-IDF 的全文搜索 + 相关性打分 + 索引管理。
- **看代码学架构**：
  - 每个服务都严格分离：`index.ts`（启动服务器）、`handlers.ts`（纯函数处理请求 + 验证）、`store.ts`（业务逻辑 + CRUD）。
  - 共享 `shared/` 里的类型、HTTP 工具、MemoryStore 基类、验证器。
  - **纯函数设计**：测试时不需要真正启动服务器，直接调用 `handleRequest()` 就能测。

#### 2. **实战练习与测试（强烈推荐）**
- 运行完整测试套件：看到 516 个测试闪绿，很有成就感。
- 修改业务规则后跑测试，看哪里会破（很好的 TDD 练习）。
- 学习如何写高质量测试：单元、业务规则、边界 case、集成测试（完整用户旅程：注册→下单→支付→通知→分析）。

#### 3. **作为 AI Agent 的实验平台**
- Hermes 子进程 Agent、General/Explore 等可以直接操作这些服务。
- 你可以让 Agent 扩展新功能、修复 bug、增加新服务，然后用 516 测试回归验证。
- 这正是我们“co-evolution”（共同进化）的 concretization：哲学上的“可控、不失控”在这里变成可运行的代码。

#### 4. **快速开始（复制粘贴就能玩）**

```bash
# 1. 安装依赖（如果还没装）
bun install

# 2. 运行全部测试（推荐先看这个）
bun services/run-tests.ts

# 3. 运行所有测试（更详细）
bun test services/

# 4. 启动单个服务（带热重载，推荐）
bun --watch services/auth/index.ts
# 然后用 curl 或 Postman 测试 http://localhost:3001

# 启动其他服务同理（端口见下面）
```

**各服务端口**：
- auth: 3001
- users: 3002
- products: 3003
- orders: 3004
- payments: 3005
- notifications: 3006
- analytics: 3007
- search: 3008

#### 推荐学习路径
1. 先跑 `bun services/run-tests.ts` 看输出。
2. 读 `services/shared/` 理解基础架构。
3. 挑一个服务（建议从 **orders** 或 **payments** 开始，看状态机和业务规则）。
4. 试着改一个规则，再跑测试看效果。
5. 最后看 `services/integration/` 里的端到端测试，理解服务间如何协作。
