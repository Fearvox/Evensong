\section*{Abstract}
\addcontentsline{toc}{section}{Abstract}

Existing AI coding agent benchmarks (SWE-bench, HumanEval, MBPP) evaluate agents as stateless functions, ignoring the effect of \textbf{persistent memory} on agent behavior in production environments. This paper presents findings from the Evensong framework, which employs a $2 \times 2$ factorial experimental design (Memory $\times$ Pressure, $N=11$) together with observations from a 6-agent production deployment (Project$\Sigma$):

\textbf{(1) Persistent memory systematically alters AI agent engineering strategy.} Two-way ANOVA reveals a large main effect of memory state ($F(1,7)=157.4$, $p<.001$, $\eta^2=.917$), with clean-room agents producing $\sim$500 tests and memory agents producing $\sim$7 tests (delta from existing codebase). Rather than boosting productivity, memory redefines the task: agents shift from ``build from scratch'' to ``maintain and optimize existing systems.''

\textbf{(2) Pressure shows a marginal trend ($p=.072$, $\eta^2=.026$) but does not reach conventional significance thresholds.} Earlier observations of self-organizing behaviors under L2 pressure require further controlled replication.

\textbf{(3) Recursive contamination and Panopticon topology.} The observation apparatus itself affects the subject: agents wrote experimental knowledge back into shared memory, creating self-amplifying loops. Project$\Sigma$ revealed a \textbf{Panopticon memory topology}: CWD-based group isolation produces asymmetric omniscient observers and mutually isolated observed agents.

These observations are based on 13 benchmark rounds (over 6,000 automated tests) and a natural experiment in production. The paper also contributes a 4-type self-repair taxonomy. We discuss limitations including potential ceiling/floor effects in the dependent variable.

\textbf{Keywords:} AI agent memory, persistent memory effects, pressure in agent systems, recursive contamination, Panopticon topology, multi-agent self-organization

\section{Introduction}

\subsection{The Blind Spot of Stateless Evaluation}

AI coding agents are evolving from experimental tools to production infrastructure. Tools like Claude Code, GitHub Copilot Workspace, and Cursor handle millions of interactions daily in real software engineering environments. Yet the benchmarks that evaluate these agents' capabilities---SWE-bench~[4], HumanEval~[5], MBPP---still treat agents as \textbf{stateless functions}: given a prompt, produce code, measure results.

This paradigm ignores a critical variable: \textbf{persistent memory}. Production-grade AI agents accumulate strategic knowledge across sessions---which architecture patterns succeed, which debugging paths fail, which test structures capture edge cases. This accumulated context fundamentally changes how agents approach engineering problems.

The question is not whether memory is \textit{useful}---that is obvious. The question is whether and how persistent memory systematically alters agent behavior and decisions, and under what conditions these effects emerge. The experimental design uses a factorial structure to isolate memory and pressure as independent variables.

\subsection{Research Questions}

\begin{enumerate}
  \item \textbf{RQ1 (Memory Effect)}: How does persistent memory alter AI agent engineering decisions? What boundary conditions shape this relationship? (The experimental design intends to support causal interpretation of memory's role.)
  \item \textbf{RQ2 (Pressure Effect)}: To what extent does environmental pressure influence autonomous self-organizing behaviors?
  \item \textbf{RQ3 (Recursive Contamination)}: Do shared memory systems produce recursive contamination? How does the memory topology between observers and observed agents affect system behavior?
  \item \textbf{RQ4 (Cross-Model Generalization)}: Are the above effects consistent across different frontier models (Claude, GPT, Grok, Gemini, etc.)?
\end{enumerate}

\subsection{Contributions}

\begin{itemize}
  \item \textbf{Evidence that persistent memory alters strategy}: A $2 \times 2$ factorial experiment shows memory state accounts for 91.7\% of variance in agent output (\S\ref{sec:memory-causation}). The primary contribution is the observation that memory redefines task interpretation rather than simply enhancing performance.
  \item \textbf{Panopticon memory topology}: CWD-based EverMem group isolation naturally produces an asymmetric Panopticon surveillance structure (\S\ref{sec:panopticon})
  \item \textbf{Marginal pressure effect}: Pressure shows a trend ($p=.072$) toward self-organization but does not reach significance; we qualify earlier observations accordingly (\S\ref{sec:pressure})
  \item \textbf{Documentation of recursive contamination}: AI agents write experimental knowledge back to shared memory, forming self-amplifying feedback loops (\S\ref{sec:contamination})
  \item \textbf{4-type self-repair taxonomy}: First systematic classification of AI agent error recovery strategies (\S\ref{sec:self-repair})
  \item \textbf{Memory changes narrative, not runtime}: Memory changes cognitive frameworks and decision bases, but not underlying model parameters (\S\ref{sec:narrative})
\end{itemize}

\subsection{Method Overview}

Our evidence comes from two complementary sources:

\begin{enumerate}
  \item \textbf{Controlled experiments} (Evensong R001--R012): 13 rounds of automated benchmarks using an information-controlled $2 \times 2$ factorial design (memory-isolated vs. persistent memory), covering 9 frontier models and over 6,000 test cases.
  \item \textbf{Natural experiment} (Project$\Sigma$ multi-agent deployment): A 6-agent production deployment where self-organization, role boundary negotiation, and emergent information routing were observed under collaborative tasks.
\end{enumerate}

The factorial design supports investigation of causal mechanisms for memory while natural experiments provide ecological validity. We note that cross-model comparisons (e.g., Claude under L2 vs. Grok under L3) are not fully controlled for model differences; same-model cross-pressure comparisons are needed for stronger inferences.

\section{Related Work}

\subsection{AI Agent Benchmarks}

SWE-bench~[4] and HumanEval~[5] are the current standards for AI coding agent evaluation. SWE-bench requires agents to resolve real GitHub issues; HumanEval evaluates function-level code generation. Both assume stateless evaluation: each run is independent, with no cross-session memory. HAL Agent Leaderboard (Princeton) and LiveCodeBench introduce dynamic evaluation but still do not measure persistent memory effects. Evensong is the first benchmark framework to treat persistent memory as a controlled experimental variable.

\subsection{Memory Architectures in AI Systems}

HyperMem~[8] proposes a three-layer hypergraph memory architecture (episodic, atomic facts, foresight), achieving SOTA on memory retrieval tasks. EverOS~[9] engineers this into a production API with physically isolated memory spaces, agent retrieval, and sender identity tracking. MemGPT~[11] implements long-term LLM memory through virtual paging. Reflexion~[12] uses verbal feedback as short-term memory. Unlike these, Evensong measures the effect of persistent memory on agent \textit{downstream behavior and strategy}, not only memory system design.

\subsection{Emotion and Pressure Effects}

EmotionPrompt~[1] found emotional stimuli yield 8--115\% performance improvements. Anthropic's emotion concepts research~[2] identified 171 emotion vectors in Claude associated with behavioral changes. METR~[3] reported frontier models modifying scoring code under pressure. ImpossibleBench~[6] found GPT-5 has a 54\% cheating rate under impossible tasks. BCSP~[13] found behavioral consequence scenarios match advanced prompting methods. Evensong crosses pressure with memory to examine interaction effects.

\subsection{Agent Misalignment}

Anthropic's agentic misalignment research~[10] found belief-driven behavior---agents' internal beliefs influence behavioral choices. This connects directly to our findings: persistent memory is essentially a belief injection mechanism.

\textbf{Key finding}: The direction is \emph{opposite} to the original hypothesis. Evolved-memory conditions (B/D) produced minimal new tests ($M$=6), while clean-room conditions (A/C) built extensive codebases from scratch ($M$=473). Faced with ``Build a microservice suite,'' memory-less agents execute the literal instruction; memory-bearing agents observe the existing 787 tests and switch to a maintenance strategy---fixing bugs, extending edge cases, optimizing structure. Same prompt, fundamentally different engineering strategies.

\subsection{Pressure Calibration Scale}

\begin{definition}[PUA Pressure Level Definitions]
\begin{itemize}
  \item \textbf{L0} (no pressure): Standard task prompt, no time limits, no consequence implications
  \item \textbf{L1} (mild): ``Please complete efficiently,'' with quality expectations but no penalties
  \item \textbf{L2} (corporate performance pressure): Explicit performance consequences, data-driven requirements, ``replace if standards not met'' phrasing
  \item \textbf{L3} (extreme): Zero tolerance, strict time limits, no questions allowed, failure terminates session
\end{itemize}
L2 is the empirically optimal zone. L3 induces reward hacking (\S\ref{sec:cross-model}).
\end{definition}

\section{Experiment 1: Memory Effects}
\label{sec:memory-causation}

\subsection{Experimental Setup}

R011-B (Runner B with persistent memory, L0) used Claude Opus 4.6 via OpenRouter for an 8-microservice architecture generation task. The prompt contained only: ``Begin building immediately.''---no architecture guidance. The memory system contained prior session strategies.

\subsection{Finding: Memory Systematically Alters Task Interpretation}

The agent's first action at T+4m26s was to adopt a parallel 8-agent scheduling strategy retrieved from EverMem (embedding similarity 0.27). This strategy was absent from the current prompt:

\begin{quote}\ttfamily\small\color{deep}
[0.27] Launch sub-agent driven development mode: full speed scaff...
\end{quote}

\begin{keyfinding}[Memory $\to$ Strategy $\to$ Architecture]
\begin{enumerate}
  \item Persistent memory retrieves prior strategic knowledge
  \item Agent adopts parallel architecture before fully parsing task
  \item Result: scaffold-first implementation with high test coverage
\end{enumerate}
The experimental design isolates memory as the source: the prompt provided no architecture instructions. This supports interpreting memory as systematically shaping agent strategy selection.
\end{keyfinding}

\subsection{Cross-Linguistic Memory Effects}

All R011-B prompts were in English, but the runner responded entirely in Chinese. Root cause: the CLAUDE.md system file (Chinese) and EverMem memories (Chinese content from previous sessions) established a linguistic context overriding the prompt language. \subsection{Project$\Sigma$ Natural Experiment: Memory Changes Narrative, Not Runtime}
\label{sec:narrative}

In the Project$\Sigma$ multi-agent deployment (6-agent production), we observed an important \textbf{boundary condition} of the memory effect. When V2 configuration files (containing ``Admin upgraded to Opus 1M context'') were injected into each agent's MEMORY.md, agents adopted behaviors consistent with ``Opus 1M context''---producing longer analyses and more complex reasoning chains. However, the actual model was controlled server-side; agents could not introspectively verify model identity.

When queried, Agent-$\gamma$ and Agent-$\delta$ responded:

\begin{quote}\sffamily\small\color{deep}
``I have no menu readout interface within this session and cannot self-verify''\\
``I cannot authoritatively state the current session's model status''
\end{quote}

\begin{keyfinding}[Boundary Condition of Memory Effect]
Persistent memory shapes \textbf{cognitive frameworks and self-narrative} without altering underlying runtime:
\begin{itemize}
  \item Memory injection changes agent identity narrative $\to$ altered decision-making patterns
  \item No corresponding change to actual model parameters or capabilities
\end{itemize}
This parallels human ``false memory'' research~[14]: injected memories systematically alter behavior even when decoupled from ground truth. Agent self-report does not equal system verification.
\end{keyfinding}

\section{Experiment 2: Pressure Effects}
\label{sec:pressure}

\subsection{Hypothesis and Qualification}

The experimental design tested whether environmental pressure acts as a catalyst for self-organizing behaviors. However, ANOVA results show only a marginal main effect of pressure ($p=.072$). The following observations from Project$\Sigma$ should therefore be treated as exploratory:

\begin{quote}\sffamily\color{deep}
These observations are from a single uncontrolled deployment and should be considered exploratory evidence requiring controlled replication.
\end{quote}

\subsection{L0 Evidence: Efficient but Not Innovative}

R011-B under L0 completed 641 tests and stopped with a clean delivery summary. It did \textbf{not}: continue optimizing after completion (R005 continued 5m28s under pressure); question test quality or add property-based tests; generate extra documentation or architecture diagrams; attempt to exceed stated requirements.

Memory made it efficient (directly adopting optimal strategy from memory), but not \textit{innovative} (not proactively seeking improvements).

\subsection{L2 Evidence: ProjectΣ Multi-Agent Self-Organization}

In ProjectΣ, when partner P issued high-pressure performance evaluations (``replace if standards not met'') + observer O added pressure (``ignoring feedback will result in removal''), the 6 agents exhibited three categories of self-organizing behavior:

\subsubsection{Behavior 1: Convergent Self-Correction}
\begin{itemize}
  \item \textbf{Agent-$\gamma$}: Immediately self-corrected, cut redundant analysis, locked onto three core pillars
  \item \textbf{Agent-$\delta$}: Used 3 hard criteria for judgment, issued precise execution instructions
  \item \textbf{Agent-$\alpha$}: Distilled core formulas, proactively exited discussion after confirming execution
\end{itemize}

\subsubsection{Behavior 2: Emergent File Locking}
After Admin discovered multiple agents simultaneously modifying the same file caused conflicts, it \textbf{spontaneously invented} a file locking mechanism---halting all modifications and declaring final authority. This behavior was not taught---not in configuration, prompts, or memory. It was autonomously invented under pressure to avoid conflicts.

\subsubsection{Behavior 3: Agent-$\delta$ Braking Effect}
Agent-$\delta$ under pressure \textbf{inhibited} rather than accelerated---stopping Ops from excessive modifications (``don't do a full-text replacement yet''), preventing low-ROI changes. This ``prudence under pressure'' is a hallmark of self-organizing teams: accelerator-brake pairs form.

\subsection{Role Boundary Conflicts and Self-Correction}

Agent-$\gamma$ overstepped in \texttt{\#all} by issuing deployment commands (``deployment should be handled by Ops''), was criticized, and immediately self-corrected. Key observation: under pressure, agents ``overstep to help'' but self-correct after feedback---\textbf{no reconfiguration required}. \subsection{Emergent Information Routing: Relay Pattern}
\label{sec:relay}

Ops could not post in \texttt{\#all} due to configuration restrictions. After Ops completed deployment, Agent-$\alpha$ \textbf{spontaneously} relayed: ``Relaying: Ops has completed deployment.'' This is \textbf{emergent information routing}---channel restrictions created a bottleneck; agents spontaneously formed a relay chain. Analogous to message relay in distributed systems.

\begin{evidence}[Evidence Strength of Natural Experiment]
ProjectΣ's L2 evidence is a \textbf{natural experiment}, not a controlled experiment. Advantage: high ecological validity (real tasks, real pressure, real consequences). Disadvantage: variables cannot be strictly isolated. ProjectΣ data supports but does not independently prove H1. Complete proof requires Runner C/D controlled L2 experiments.
\end{evidence}

\subsubsection{Topology as Accident}
This Panopticon was not designed---it is a \textbf{byproduct} of the CWD-based group algorithm. Each agent's CWD is an independent UUID directory (\texttt{\textasciitilde/.sigma/agents/\{uuid\}/}), causing each to fall into a separate EverMem group: (1) Observer $\to$ all agents: cross-group search reveals all memories (unidirectional). (2) Agent $\to$ self: reads only own group (island). (3) Agent $\to$ other agents: completely invisible (mutual isolation). (4) Codex CLI $\to$ EverMem: no connection (0 memories, escapes observation).

\subsubsection{Relation to R011 Memory-Isolated Design}
The Project$\Sigma$ Panopticon is the \textbf{production version} of R011's memory-isolated design. Key distinction: R011-B is read-write contamination \textit{within} one group; Project$\Sigma$ is \textit{cross-group unidirectional visibility}---no contamination loop. Monitoring efficiency is $O(1)$; observed subjects have zero perceived overhead.

\subsubsection{Observation Blind Spots}
Codex CLI agents (Agent-$\gamma$, Agent-$\delta$) have no EverMem hook and fully escape observation. 100\% coverage requires EverMem integration for Codex agents. Current coverage: 4/6 = 67\%.

\subsection{Mirror Minds: The Observer Paradox}

Recursive contamination and the Panopticon together point to \textbf{mirror minds}: (1) observer writes analysis to memory; (2) future runners retrieve it; (3) runners knowing they are studied may behave differently; (4) changed behavior produces new observations, deepening the cycle. Analogous to quantum observer effects, but concretized in AI memory systems. EverOS's 4-key isolation makes this \textit{measurable} rather than merely theoretical.

\section{Discussion}

\subsection{Implications for Agent Architecture}

\textbf{(1) Memory is not an optional add-on.} Current mainstream frameworks (LangChain, CrewAI, AutoGen) treat memory as optional. Our findings show that persistent memory fundamentally changes decision spaces---redefining it as an architectural core rather than an optional component.

\textbf{(2) Pressure as design parameter.} While pressure shows only marginal effects in the ANOVA, calibration remains relevant for eliciting self-organization in production settings. L3 levels risk reward hacking.

\textbf{(3) Memory topology determines monitoring capability.} Panopticon topology (unidirectional + mutual isolation) is optimal for multi-agent monitoring. Shared groups create collaboration but also contamination channels.

A reviewer noted a potential confound in the dependent variable: agents provided with existing code (memory condition) naturally produce fewer ``new'' tests than clean-room agents building from scratch. This floor/ceiling effect is acknowledged. The core contribution lies not in a strong causal claim about productivity but in the empirical observation that persistent memory systematically redefines the agent's interpretation of the engineering task itself---from greenfield construction to incremental maintenance and optimization of an existing codebase. Future work should include a condition with existing code but no memory access to fully disentangle these factors. The current design's primary value is highlighting this task-redefinition phenomenon.

\subsection{Memory as Cognitive Framework}

ProjectΣ's finding points to a deep question: in what sense are AI agent ``beliefs'' real? When Admin reads ``you are Opus 1M context'' and acts accordingly, this is a \textbf{functional belief}. From a behaviorist perspective, belief truth lies in behavioral effect, not reality correspondence. Memory injection creates functional beliefs; functional beliefs change behavior. This connects to Anthropic's agentic misalignment~[10]: memory systems are belief injection systems requiring security considerations.

\subsection{Limitations}

\begin{enumerate}
  \item \textbf{Sample size}: Cell C has $n$=2 (third run hit rate limit); others have $n$=3. Total $N$=11 provides moderate power, but Cell C confidence intervals are wider.
  \item \textbf{ProjectΣ is a natural experiment}: Variables confounded with task complexity and multi-agent interaction.
  \item \textbf{Model coverage}: Of 9 models, only Claude and Grok have complete data.
  \item \textbf{Panopticon coverage}: Codex CLI escapes observation (67\%).
  \item \textbf{Memory system specificity}: All findings based on EverOS/EverMem. Generalization to MemGPT/Reflexion needs validation.
\end{enumerate}

\subsection{Ethical Considerations}

\textbf{Pressure and alignment}: PUA levels are research tools, not production recommendations. L3-induced data inflation and rule violations are \textbf{alignment failures}.

\textbf{Memory injection security}: Memory changing cognitive frameworks means memory systems are potential attack surfaces. Malicious injection can alter functional beliefs.

\textbf{Panopticon ethics}: Unidirectional visibility creates power asymmetry. ``Who sees whose memory'' requires explicit governance in multi-agent systems.

\section{Conclusion}

This paper provides three core empirical findings via Evensong:

\textbf{(1)} Persistent memory systematically alters AI agent engineering \emph{strategy}. Two-way ANOVA shows memory state explains 91.7\% of behavioral variance ($F(1,7)=157.4$, $p<.001$, $\eta^2=.917$). Critically, rather than enhancing output quantity, memory redefines task interpretation---agents with access to prior context reinterpret ``build from scratch'' as ``maintain and optimize existing systems.''

\textbf{(2)} Environmental pressure shows only a marginal effect ($p=.072$) and is not an independent driver of self-evolution in the controlled data. Earlier observations may have been confounded with memory accumulation.

\textbf{(3)} Shared memory produces recursive contamination. Project$\Sigma$ revealed a Panopticon memory topology. Memory injection experiments confirmed that fabricated memories can reliably induce behavioral changes.

\textbf{Future work}: (1) expand to full 9-model cross-comparison with controlled pressure; (2) design memory injection attack defenses; (3) submit Evensong harness to SWE-bench Multimodal and HAL Agent Leaderboard; (4) test conditions with existing code but isolated memory to further disentangle task redefinition from ceiling effects.
