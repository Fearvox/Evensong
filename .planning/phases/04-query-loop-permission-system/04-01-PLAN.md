**测试覆盖（PERM-01 + PERM-02）：**

**PERM-01 — 权限在执行前检查（非 bug，写通过测试）：**
在测试文件 describe 块顶部添加一条明确注释：
```
// PERM-01: Research verified that resolveHookPermissionDecision() in toolExecution.ts
// runs at line ~921 BEFORE tool.call() at line ~1010+.
// The test below proves hasPermissionsToUseTool() returns 'deny' before any tool.call()
// can fire — no code fix needed, this is the passing test marking PERM-01 resolved.
```
写一个测试：构造 deny rule → 调用 hasPermissionsToUseTool → 断言 behavior === 'deny'（tool.call 永远不会被调用，因为函数在权限拒绝时提前返回）。

**PERM-02 — 三种模式正确执行：**

1. `alwaysDenyRules` 测试：
   - 构造 `createTestToolUseContext({ getAppState: () => ({ toolPermissionContext: { ...getEmptyToolPermissionContext(), alwaysDenyRules: { session: ['TestTool'] } } } as AppState) })`
   - 调用 `hasPermissionsToUseTool(mockTool, {}, ctx, mockParentMsg, 'tu-1')`
   - 断言 `result.behavior === 'deny'`

2. `alwaysAskRules` 测试：
   - 同上但用 `alwaysAskRules: { session: ['TestTool'] }`
   - 断言 `result.behavior === 'ask'`

3. `bypassPermissions` 模式测试：
   - 构造 `toolPermissionContext` 中 `mode: 'bypassPermissions'`（或 `'auto'` — 读实际代码确认 bypass 的 mode 值）
   - 断言 `result.behavior === 'allow'`

4. 无规则时默认 passthrough（非 deny 工具的读取模式）：
   - 用 `getEmptyToolPermissionContext()`（无任何规则）
   - 用一个 `checkPermissions` 返回 `{ behavior: 'passthrough' }` 的 mock 工具
   - 断言 `result.behavior === 'ask'`（passthrough → ask）

**Mock 工具构造：**
不要使用 `buildTool()`（复杂依赖）。直接构造符合 Tool 接口的最小 mock 对象，包含：
- `name: 'TestTool'`
- `inputSchema: { parse: (x: unknown) => x }`（简单 passthrough）
- `checkPermissions: async () => ({ behavior: 'passthrough' as const, message: '' })`
- `isReadOnly: () => true`
- `isConcurrencySafe: () => true`
- 其他必要字段根据实际 Tool 类型补充（读 Tool.ts 确认）

如果 `hasPermissionsToUseTool` 需要 assistantMessage 且访问 `assistantMessage.message.id`（analytics 路径），使用 BashTool.test.ts 的 `mockParentMessage` shape 或扩展它。

所有测试使用 `describe` + `test` 结构，从 `bun:test` import。
