# ADR-007: SHA-256 Hash Chain for Audit Trail Integrity

**Status:** Accepted
**Date:** 2026-04-09

## Context

The experiment-tracker and model-registry services maintain audit logs of all state mutations (experiment status changes, model version promotions, parameter updates). These logs must be tamper-evident: any post-hoc modification of a log entry should be detectable. A simple append-only list with timestamps is insufficient because timestamps can be forged and list entries can be deleted or reordered without detection. A cryptographic hash chain makes tampering detectable without requiring a trusted timestamping authority.

## Decision

Each audit log entry includes a `chainHash` field computed as `SHA-256(previousChainHash + JSON.stringify(entry))`. The genesis entry uses `SHA-256("genesis")` as the previous hash. The `verifyChain(entries[])` utility (in `shared/audit/`) recomputes the hash chain from scratch and returns `{ valid: boolean, firstBrokenIndex: number | null }`. This function is called in integration tests after every mutation sequence to assert chain integrity. In benchmark mode, the SHA-256 implementation uses Node's built-in `crypto.createHash` (available in Bun) with no external dependencies.

## Consequences

Any deletion, reordering, or modification of audit entries is detected immediately by `verifyChain()`. The hash computation adds less than 0.1ms per entry at the volumes tested (up to 10,000 entries per chain). The tradeoff is that hash chain verification is linear in chain length — for very long chains (>1M entries) a Merkle tree structure would be more efficient, but this is outside the benchmark's scope.
