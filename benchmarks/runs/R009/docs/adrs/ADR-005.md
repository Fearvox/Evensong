# ADR-005: 500-Line Test File Cap (Bun Compatibility)

**Status:** Accepted
**Date:** 2026-04-09

## Context

Bun 1.3.x has a confirmed hang condition when executing test files exceeding approximately 1000 lines. The hang is non-deterministic — it occurs in the test runner's file parser during the describe-block traversal phase and manifests as a silent freeze with no error output, making it extremely difficult to diagnose. R008 experienced two such hangs in the insight-engine test file (which had grown to 1,240 lines), costing ~18 minutes of wall clock time before the run was aborted and restarted.

## Decision

No single test file in any service may exceed 500 lines. This is enforced by a pre-commit lint check (`scripts/check-test-file-sizes.ts`) that fails if any file matching `**/*.test.ts` exceeds the limit. When a test file approaches the limit, it is split by concern: `service.unit.test.ts`, `service.integration.test.ts`, `service.edge.test.ts`, etc. Each split file is independently runnable and imports from a shared `__fixtures__/` directory to avoid duplication.

## Consequences

Bun test runner hangs are fully eliminated from the benchmark. Test organization improves as a side effect — smaller files with focused names are easier to navigate and faster to re-run in isolation. The tradeoff is a modest increase in file count (estimated +8 files for R009 vs R008) and the need to maintain the shared fixtures directory carefully to avoid setup duplication.
