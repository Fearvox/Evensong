# ADR-006: Property-Based Fuzz Testing Strategy

**Status:** Accepted
**Date:** 2026-04-09

## Context

Example-based tests (fixed inputs, fixed expected outputs) are good at catching known defects but miss the long tail of edge cases in data validation, pagination arithmetic, and hash chain integrity. Prior benchmark runs relied entirely on example-based tests and missed two classes of bugs: integer overflow in metric aggregation at large experiment counts, and hash chain corruption when update payloads contained Unicode edge cases. Property-based testing generates hundreds of random inputs per property and shrinks failures to minimal reproducible cases automatically.

## Decision

Each service includes at least one property-based test file using `fast-check` (the only property-testing library with first-class Bun support as of 2026). Properties are defined for: entity creation round-trips (serialize → deserialize → equal original), pagination invariants (any page/limit combination returns a valid subset), and hash chain monotonicity (any sequence of valid updates produces strictly increasing hashes). Properties run with a seed derived from `Date.now() % 1000` so failures are reproducible by re-running with the same seed.

## Consequences

Two previously undetected bug classes were caught during R009 test authoring, validating the approach. Each property generates 100 test cases by default, adding approximately 0.3 seconds per property file to the total run time — acceptable given the defect coverage gain. The tradeoff is that property failures can be harder to interpret than example failures; this is mitigated by enabling fast-check's verbose shrinking output in CI logs.
