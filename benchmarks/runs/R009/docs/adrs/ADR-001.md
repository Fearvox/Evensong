# ADR-001: In-Memory Stores for Benchmark Runs

**Status:** Accepted
**Date:** 2026-04-09

## Context

The benchmark suite must spin up 10 services, execute hundreds of test cases, and tear down cleanly within a 15-minute wall clock budget. Running real PostgreSQL migrations, seeding persistent schemas, and managing transactional state across test runs introduces setup latency, flaky teardown, and cross-test contamination risks. The benchmark goal is to validate service logic and inter-service contracts, not database migration correctness.

## Decision

All services use in-memory Maps and arrays as their primary data store during benchmark execution. A `resetStore()` function is exported from each service's store module and called in `beforeEach` / `afterEach` hooks to guarantee isolation. Postgres, Redis, and MinIO containers are still started (to validate connection handling and health checks) but are not used as the authoritative data layer in unit or integration tests.

## Consequences

Test setup time drops from ~8 seconds (with real migrations) to under 200ms per suite. Cross-test state leakage is eliminated by design. The tradeoff is that SQL query logic, index performance, and migration correctness are not exercised by the benchmark — those concerns are deferred to a dedicated persistence regression suite outside the benchmark window.
