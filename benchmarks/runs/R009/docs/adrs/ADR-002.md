# ADR-002: Event Bus Pattern for Cross-Service Communication

**Status:** Accepted
**Date:** 2026-04-09

## Context

Ten services need to react to lifecycle events produced by other services — for example, the paper-engine needs to know when an experiment completes, and the insight-engine needs training-pipeline progress events for anomaly detection. Direct HTTP calls between all service pairs would create a dense coupling graph (up to 90 directed edges) that is fragile under partial failure and hard to extend. An alternative is a shared event bus that services publish to and subscribe from independently.

## Decision

All cross-service communication uses RabbitMQ as the event bus via a shared `@evensong/event-bus` package. Services publish typed events to named exchanges (`experiment.events`, `training.events`, `review.events`, etc.) and declare their own queues with explicit routing keys. The shared package exports strongly-typed `publish<T>()` and `subscribe<T>()` helpers that wrap the amqplib channel. In benchmark mode, the event bus is replaced with a synchronous in-process `EventEmitter` stub exported from `@evensong/event-bus/mock` to avoid AMQP connection overhead in unit tests.

## Consequences

Services are fully decoupled: adding a new consumer requires zero changes to the producer. The mock bus enables deterministic unit testing without a running RabbitMQ broker. The tradeoff is that event schema drift is not caught at compile time — this is mitigated by the shared typed event definitions in `shared/events/` which are imported by both producer and consumer, and validated with Zod at runtime boundaries.
