# ADR-004: Two-Wave Agent Dispatch for Parallel Builds

**Status:** Accepted
**Date:** 2026-04-09

## Context

R008 showed that dispatching all 10 service build agents simultaneously saturated the available CPU and memory, causing the insight-engine (the heaviest service) to hang mid-build and blow the wall clock budget. Sequential dispatch is safe but slow (~3.5 minutes per service = 35 minutes total). A smarter strategy is needed to maximize parallelism while preventing resource contention.

## Decision

Services are dispatched in two waves based on their dependency graph. Wave 1 dispatches the four infrastructure-independent leaf services (experiment-tracker, model-registry, dataset-vault, auth-gateway) simultaneously. Wave 2 begins only after all Wave 1 agents report completion, then dispatches the remaining six services (training-pipeline, paper-engine, compute-scheduler, review-system, collab-hub, insight-engine) which depend on Wave 1 outputs. Each wave uses `Promise.allSettled()` so a single failure does not cancel sibling builds. The insight-engine is always placed last in Wave 2 and given a dedicated 90-second timeout.

## Consequences

Total build time drops from ~35 minutes (sequential) to ~12 minutes (two-wave), matching R007 performance. The explicit wave boundary prevents the memory spike that crashed R008. The tradeoff is that Wave 2 cannot start until the slowest Wave 1 service finishes, introducing a small synchronization wait — acceptable given the overall wall clock improvement.
