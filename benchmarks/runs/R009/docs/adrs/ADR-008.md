# ADR-008: Z-Score Anomaly Detection Threshold Selection

**Status:** Accepted
**Date:** 2026-04-09

## Context

The insight-engine performs anomaly detection on experiment metrics (loss curves, accuracy deltas, training throughput) to surface outlier runs that may indicate bugs, hardware failures, or unexpectedly strong results. The threshold determines the false-positive / false-negative tradeoff: too low a threshold generates noise alerts that erode researcher trust; too high misses genuine anomalies. Standard choices in the literature are Z ≥ 2.0 (liberal), Z ≥ 2.5 (moderate), and Z ≥ 3.0 (conservative). The choice must be validated against the benchmark's synthetic experiment dataset.

## Decision

The insight-engine uses a Z-score threshold of **2.5** for all continuous metrics. This was selected after running the detection algorithm against R008's 664-test synthetic dataset and measuring precision/recall: Z=2.0 produced 14% false positive rate (too noisy), Z=3.0 produced 22% false negative rate (too permissive for the loss curve use case), and Z=2.5 achieved 4% FPR and 8% FNR — the best balance for the expected experiment volume. The threshold is configurable via `ANOMALY_ZSCORE_THRESHOLD` env var to support future tuning without code changes.

## Consequences

Researchers receive alerts for genuine anomalies with low noise at the benchmark's synthetic data distribution. The env var override allows threshold adjustment per deployment context without redeployment. The tradeoff is that Z=2.5 assumes a roughly normal distribution of metric values — for heavily skewed metrics (e.g. training time on heterogeneous hardware), a percentile-based approach would be more robust, which is noted as a future improvement in the insight-engine backlog.
