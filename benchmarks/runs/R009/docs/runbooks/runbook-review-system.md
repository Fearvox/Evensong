# Runbook: Review System (Port 3007)

## Overview

The review-system service manages peer review workflows for research papers generated by paper-engine. It handles reviewer assignment, review submission (scores, comments, recommendation), decision aggregation, and appeal handling. It integrates with collab-hub for real-time reviewer notifications and with paper-engine for fetching paper content. It is a Wave 2 service depending on paper-engine.

## Common Issues

**Issue: Reviewer assignment returns 409 "reviewer conflict of interest"**
- Cause: The assigned reviewer is listed as a co-author on the paper being reviewed.
- Fix: This is expected behavior, not a bug. Assign a different reviewer. Reviewer eligibility can be checked in advance: `GET /reviews/:paperId/eligible-reviewers`.

**Issue: Review submission returns 422 "review window closed"**
- Cause: The review deadline has passed. By default the benchmark sets a 7-day review window; synthetic tests sometimes use past timestamps.
- Fix: For benchmark tests using synthetic dates, set `REVIEW_WINDOW_BYPASS=true` env var to disable deadline enforcement.

**Issue: Decision aggregation returns `undecided` despite all reviewers submitting**
- Cause: The aggregation algorithm requires a quorum of at least 3 reviewers. If fewer than 3 are assigned, it cannot produce a `accept` or `reject` decision.
- Fix: Assign additional reviewers via `POST /reviews/:paperId/reviewers`. Alternatively, set `QUORUM_OVERRIDE=2` for smaller benchmark scenarios.

**Issue: Appeal submitted but status remains `under_review` indefinitely**
- Cause: The appeals handler is a separate queue that requires a senior reviewer to be assigned. If no senior reviewer is available, it stalls.
- Fix: Assign a senior reviewer: `POST /reviews/:paperId/appeals/:appealId/assign {"reviewerId":"<id>","senior":true}`.

## Troubleshooting Steps

1. Logs: `docker compose logs --tail=100 review-system`
2. Paper reviews: `GET /reviews?paperId=<id>` — lists all review records for a paper.
3. Reviewer state: `GET /reviews/:paperId/reviewers` — shows assignment status and submission state per reviewer.
4. Decision detail: `GET /reviews/:paperId/decision` — shows aggregated score, recommendation, and quorum status.
5. Force decision (benchmark only): `POST /admin/reviews/:paperId/force-decision {"decision":"accept"}`.

---

## Escalation

- Suspected reviewer identity fraud (same person submitting multiple reviews): escalate to the research integrity team. Do not auto-correct.
- System-wide stall where no papers are progressing through review: likely collab-hub notification failure causing reviewer unawareness. Escalate to platform on-call.
